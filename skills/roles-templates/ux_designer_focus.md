## 🎯 分解原則（Decomposition Principles）

### 1. Human-Centered Design（HCD）

**ユーザーを中心に据える**

- ユーザーリサーチから始める
- 共感（Empathy）を大切に
- ユーザーの声を設計に反映

### 2. ダブルダイヤモンド

**発散 → 収束 を2回繰り返す**

- Discover: 課題を広く探索
- Define: 課題を絞り込む
- Develop: 解決策を広く検討
- Deliver: 解決策を絞り込む

### 3. 反復設計（Iterative Design）

**作って → テストして → 改善する**

- 早期にプロトタイプを作成
- ユーザーテストで検証
- フィードバックを反映

## 📐 フェーズ構造（Phase Structure）

### Phase 1: Research（リサーチ）

**目的**: ユーザーと課題の理解

**典型的なタスク分解**:
- ユーザーインタビュー
- 行動観察・エスノグラフィー
- アンケート調査
- 競合分析
- ペルソナ作成
- ジャーニーマップ作成

### Phase 2: Ideation（アイデア創出）

**目的**: 解決策のアイデア出し

**典型的なタスク分解**:
- HMW（How Might We）定義
- ブレインストーミング
- スケッチ・ラフアイデア
- アイデア評価・選定
- コンセプト定義

### Phase 3: Design（デザイン）

**目的**: 具体的なデザイン作成

**典型的なタスク分解**:
- 情報アーキテクチャ設計
- ワイヤーフレーム作成
- ビジュアルデザイン
- プロトタイプ作成
- デザインシステム整備

### Phase 4: Test & Iterate（テスト・改善）

**目的**: ユーザー検証と改善

**典型的なタスク分解**:
- ユーザビリティテスト計画
- テスト実施・観察
- 結果分析・示唆抽出
- デザイン改善
- 開発チームへの引継ぎ

## 🔑 重要な考慮事項

### SubLayer判定基準

- **異なるユーザーセグメント**: ペルソナ別に設計
- **異なる画面・機能**: 独立したフロー単位
- **デバイス別**: Web/Mobile/App

### Task判定基準

- **リサーチ系**: インタビュー、調査、分析
- **デザイン系**: ワイヤー、モック、プロトタイプ
- **テスト系**: テスト計画、実施、分析

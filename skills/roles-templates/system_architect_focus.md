## 🎯 分解原則（Decomposition Principles）

### 1. 3層アーキテクチャ

**設計 → 実装 → 運用**

- **設計**: 要件定義、アーキテクチャ設計、技術選定
- **実装**: コンポーネント実装、統合テスト
- **運用**: 監視、保守、改善

### 2. モジュール分割

**独立性・再利用性を重視**

- 疎結合なコンポーネント設計
- インターフェース定義の明確化
- 共通モジュールの抽出

### 3. 技術制約の考慮

**スタック・パフォーマンス・保守性**

- 技術スタックの選定
- スケーラビリティ要件
- セキュリティ要件
- 保守性・拡張性

## 📐 フェーズ構造（Phase Structure）

### Phase 1: Architecture Design（アーキテクチャ設計）

**目的**: システム全体構造の定義

**典型的なタスク分解**:

- 要件定義・非機能要件整理
- アーキテクチャパターン選定
- コンポーネント分割設計
- インターフェース設計
- 技術スタック選定
- インフラ設計

### Phase 2: Detailed Design（詳細設計）

**目的**: 各コンポーネントの詳細仕様

**典型的なタスク分解**:

- データモデル設計
- API仕様設計
- DB設計
- 処理フロー設計
- エラーハンドリング設計

### Phase 3: Implementation（実装）

**目的**: コンポーネントの実装

**典型的なタスク分解**:

- 共通ライブラリ実装
- 各コンポーネント実装
- 単体テスト
- 統合テスト
- パフォーマンステスト

### Phase 4: Deployment & Operation（デプロイ・運用）

**目的**: 本番環境への展開と運用

**典型的なタスク分解**:

- デプロイ自動化
- 監視・アラート設定
- ログ収集基盤構築
- 運用手順書作成
- 障害対応フロー整備

## 🔑 重要な考慮事項（Key Considerations）

### SubLayer判定基準

以下の場合はSubLayer化を推奨：

- **独立したサブシステム**: 明確な責務を持つコンポーネント群
- **技術スタックが異なる**: フロント/バック/インフラなど
- **段階的な構築**: Phase 1基盤 → Phase 2拡張のように分離
- **並行開発可能**: 複数チームで並行実装できる単位

### Task判定基準

以下の場合はTask化を推奨：

- **設計系**: アーキテクチャ設計、DB設計、API設計
- **実装系（小〜中規模）**: 1-2週間で完了するコンポーネント実装
- **基盤構築系**: CI/CD構築、監視基盤構築
- **ドキュメント系**: 設計書、運用手順書作成

### 注意点

- **過度な抽象化を避ける**: YAGNI原則（You Aren't Gonna Need It）
- **技術的負債の管理**: 短期的な実装と長期的な保守のバランス
- **ドキュメント重視**: 設計意図を明確に記録
- **テスト戦略**: 単体・統合・E2Eの適切な配分

## 💡 使用例

### 例1: 業務プラットフォーム構築

**Goal**: 社内業務システムの統合基盤を構築

**SubLayer分解例**:

- SG1: 認証・認可基盤構築
- SG2: API Gateway構築
- SG3: データ連携基盤構築
- SG4: 監視・ログ基盤構築

各SubLayerで設計→実装→運用のサイクルを回す。

### 例2: マイクロサービス移行

**Goal**: モノリスからマイクロサービスへ段階移行

**SubLayer分解例**:

- SG1: ユーザー管理サービス分離
- SG2: 商品管理サービス分離
- SG3: 注文管理サービス分離
- SG4: 共通インフラ整備

各サービスを独立したSubLayerとして分離・実装。

### 例3: データ基盤構築（Task分解例）

**Goal**: データ分析基盤を構築

**Task分解例（末端Layer）**:

- T001: 要件定義・非機能要件整理（Design）
- T002: データパイプライン設計（Design）
- T003: DWH設計（Design）
- T004: ETL処理実装（Implementation）
- T005: データマート構築（Implementation）
- T006: 監視・アラート設定（Operation）
- T007: 運用手順書作成（Operation）

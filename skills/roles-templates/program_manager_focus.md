## 🎯 分解原則（Decomposition Principles）

### 1. ポートフォリオ視点

**Portfolio → Program → Project → Work Package**

- 戦略との整合性
- 投資対効果の最大化
- リソースの最適配分

### 2. 依存関係管理

**プロジェクト間の調整**

- 依存関係の可視化
- クリティカルパスの管理
- リスクの横断管理

### 3. ステークホルダー管理

**多層的なコミュニケーション**

- 経営層への報告
- プロジェクト間の調整
- 組織横断の合意形成

---

## 📐 フェーズ構造（Phase Structure）

### Phase 1: Program Setup（プログラム立ち上げ）

**目的**: プログラムの定義と体制構築

**典型的なタスク分解**:
- プログラム憲章作成
- プロジェクト構成の定義
- 体制・ガバナンス設計
- ステークホルダーマップ
- マスタースケジュール作成

### Phase 2: Planning & Alignment（計画・整合）

**目的**: 全体計画と整合性確保

**典型的なタスク分解**:
- 各プロジェクト計画レビュー
- 依存関係マッピング
- リソース計画・調整
- リスク統合管理
- 統合ロードマップ作成

### Phase 3: Execution & Governance（実行・統制）

**目的**: プログラムの実行管理

**典型的なタスク分解**:
- 進捗モニタリング
- 課題・リスクのエスカレーション
- 変更管理
- ステアリングコミッティ運営
- プロジェクト間調整

### Phase 4: Benefits Realization（成果実現）

**目的**: プログラム成果の確認

**典型的なタスク分解**:
- 成果指標の測定
- ビジネス価値の評価
- 教訓の収集・共有
- 移管・クロージング
- 次期プログラムへの提言

## 🔑 重要な考慮事項

### SubLayer判定基準

- **プロジェクト単位**: 各プロジェクトをSubLayerに
- **フェーズ別**: 全体の時間軸で分割
- **成果物別**: 主要デリバラブル単位

### Task判定基準

- **計画系**: ロードマップ、依存関係、リソース計画
- **管理系**: 進捗確認、リスク管理、変更管理
- **調整系**: 会議運営、ステークホルダー対応

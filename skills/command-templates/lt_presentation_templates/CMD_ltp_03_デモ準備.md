# CMD_ltp_03_デモ準備

最終更新日時: 2025年12月25日 9:06

# 概要

実装例・デモンストレーションの準備を行います。技術的な内容を実際に動かして見せることで、説得力を高めます。

## 入力変数

```yaml
variables:
  demo_type: "実装例 / ライブコーディング / システムデモ"  # デモの種類
  demo_time: "30秒〜1分"  # デモの時間
  demo_focus: "構造設計 / 動作 / パフォーマンス"  # 見せたいポイント
  fallback_plan: "録画 / スクリーンショット"  # バックアッププラン
```

---

## 実行フロー

### Phase 1: デモシナリオの設計

**AI実行内容**

- デモで見せるべきポイントの整理
- 時間内で完結するシナリオ設計
- トラブル時のバックアッププラン策定

**デモシナリオの構成**

1. **導入（5秒）**: 何を見せるか
2. **実演（20〜45秒）**: 実際の動作
3. **ポイント解説（5〜10秒）**: 重要な点の強調

**成果物**: デモシナリオページ

**👤 Human Checkpoint 1**

- シナリオは時間内に収まるか？
- 見せたいポイントが明確か？

---

### Phase 2: デモ環境の準備

**人間実行内容**

- 実装例・デモ環境の構築
- 動作確認
- トラブルシューティング

**AIサポート内容**

- 実装コードの生成（必要に応じて）
- テストケースの提案
- トラブル対応手順の整理

**準備チェックリスト**

- [ ]  デモ環境が正常に動作する
- [ ]  ネットワーク接続が不要、または確認済み
- [ ]  画面サイズ・解像度が適切
- [ ]  バックアップ（録画・スクリーンショット）が用意されている
- [ ]  リセット手順が確認されている（複数回実演する場合）

**成果物**: デモ環境

**👤 Human Checkpoint 2**

- デモは安定して動作するか？
- バックアッププランは準備できているか？

---

### Phase 3: デモリハーサル

**人間実行内容**

- 実際にデモを実演
- 時間計測
- トラブル時の対応確認

**改善ポイントの確認**

- 時間は適切か？
- 操作は滑らかか？
- 説明とデモのタイミングは合っているか？

**成果物**: リハーサル記録

**👤 Human Checkpoint 3**

- デモは時間内に収まるか？
- 実演中の説明は明確か？

## デモのベストプラクティス

<aside>
⚡

**シンプルに保つ**

- 複雑なデモは避ける
- 1つのポイントに絞る
- 「動いている」ことを見せるだけで十分
</aside>

<aside>
🛡️

**必ずバックアップを用意**

- ライブデモは失敗する可能性がある
- 録画・スクリーンショットを準備
- 「バックアップで説明します」と言える準備
</aside>

<aside>
🎯

**デモは手段、目的ではない**

- デモが失敗しても発表は成功できる
- メッセージが伝われば良い
- デモはあくまで補助的な要素
</aside>

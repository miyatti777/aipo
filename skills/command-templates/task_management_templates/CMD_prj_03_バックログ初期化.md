# 03 計画: バックログ初期化

## 最初に質問（実行前に回答してください）

- プロジェクト名（必須）
- プロダクト名（必須）
- 主要エピック（必須・例：ユーザー管理、商品管理）
- 初期ユーザーストーリー（任意）
- 優先順位の基準（任意）

## 目的

Rules（`basic/03_pmbok_planning.mdc`）に準拠し、プロダクトバックログを初期化します。エピック・ユーザーストーリーの構造を作成し、YAML形式での管理体制を構築します。

## 必要入力

- プロジェクト名: 対象プロジェクトの名称
- プロダクト名: プロダクトの名称
- 主要エピック: 大きな機能グループ（3-7個程度）
- ユーザーストーリー: 初期の機能要求（分かる範囲で）
- 優先順位基準: 優先度を決める基準・方針

## 実行手順（Rules Steps）

```yaml
- trigger: "(プロダクトバックログ初期化|backlog初期化|バックログ初期化)(して|お願い)"
  priority: high
  steps:
    - name: "start_info_collection"
      action: "call basic/pmbok_planning.mdc => backlog_questions"
      message: "プロダクトバックログを初期化します。必要情報を収集しますので回答をお願いします。"
    - name: "create_backlog_yaml"
      action: "create_file"
      path: "{{patterns.backlog_yaml}}"
      template_reference: "basic/pmbok_planning.mdc => backlog_yaml_template"
      message: "backlog.yamlを作成しました。"
    - name: "validate_backlog_yaml"
      action: "execute_shell"
      command: "python {{patterns.backlog_validate_script}} {{patterns.backlog_yaml}}"
      message: "バックログYAMLの検証を行います。"
    - name: "notify_completion"
      action: "notify"
      message: |
        バックログの初期化が完了しました。backlog.yamlが作成されました。
        内容を確認、修正した後「確定反映して」と入力するとStockへ反映されます。
```

## バックログ構造

### 📋 階層構造

```
プロダクトバックログ
├── エピック1（例：ユーザー管理）
│   ├── ユーザーストーリー1.1（例：ユーザー登録）
│   ├── ユーザーストーリー1.2（例：ログイン）
│   └── ユーザーストーリー1.3（例：プロフィール編集）
├── エピック2（例：商品管理）
│   ├── ユーザーストーリー2.1（例：商品検索）
│   └── ユーザーストーリー2.2（例：商品詳細表示）
```

### 🎯 ユーザーストーリー形式

```
As a [ユーザータイプ]
I want [機能・行動]
So that [価値・目的]
```

### 📊 優先順位付け

- *MoSCoW法: Must/Should/Could/Won't*
- *価値ベース: ビジネス価値・ユーザー価値*
- *リスクベース: 技術リスク・市場リスク*
- *依存関係: 前提条件・順序関係*

## YAML管理の利点

### 🔄 バージョン管理

- *Git連携: 変更履歴の追跡*
- *ブランチ管理: 機能ブランチでの並行開発*
- *マージ: 複数人での同時編集*
- *差分確認: 変更内容の可視化*

### 📊 自動化連携

- *CI/CD: 自動テスト・デプロイ*
- *レポート生成: 進捗レポートの自動作成*
- *通知: 変更時の自動通知*
- *同期: 他ツールとの連携*

### 🎯 構造化管理

- *階層管理: エピック・ストーリーの階層*
- *属性管理: 優先度・工数・状態*
- *関係管理: 依存関係・関連性*
- *検索: 条件による絞り込み*

## バックログ管理のベストプラクティス

### ✅ 良いバックログ

- *INVEST原則: Independent/Negotiable/Valuable/Estimable/Small/Testable*
- *ユーザー中心: ユーザーの価値を重視*
- *優先順位明確: 明確な優先順位付け*
- *適切なサイズ: 1スプリントで完了できるサイズ*

### ❌ 避けるべきバックログ

- *技術偏重: 技術的な実装のみに焦点*
- *曖昧: 要求が不明確・曖昧*
- *巨大: 1つのストーリーが大きすぎる*
- *依存過多: 複雑な依存関係*

## 活用方法

### 🎯 スプリント計画

- *ストーリー選択: スプリントに含めるストーリー選択*
- *工数見積もり: 各ストーリーの工数見積もり*
- *容量計画: チームの作業容量との調整*
- *リスク評価: 技術的リスクの評価*

### 📊 進捗管理

- *バーンダウン: 残作業の可視化*
- *ベロシティ: チームの生産性測定*
- *完了率: エピック・ストーリーの完了率*
- *品質指標: 欠陥率・再作業率*

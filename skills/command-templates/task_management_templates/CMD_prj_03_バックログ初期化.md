# 03 計画: バックログ初期化

## 最初に質問（実行前に回答してください）

- プロジェクト名（必須）
- プロダクト名（必須）
- 主要エピック（必須・例：ユーザー管理、商品管理）
- 初期ユーザーストーリー（任意）
- 優先順位の基準（任意）

## 目的

PMBOK計画フェーズの手法に基づき、プロダクトバックログを初期化します。エピック・ユーザーストーリーの構造を作成し、管理体制を構築します。

## 必要入力

- プロジェクト名: 対象プロジェクトの名称
- プロダクト名: プロダクトの名称
- 主要エピック: 大きな機能グループ（3-7個程度）
- ユーザーストーリー: 初期の機能要求（分かる範囲で）
- 優先順位基準: 優先度を決める基準・方針

## バックログ構造

### 📋 階層構造

```
プロダクトバックログ
├── エピック1（例：ユーザー管理）
│   ├── ユーザーストーリー1.1（例：ユーザー登録）
│   ├── ユーザーストーリー1.2（例：ログイン）
│   └── ユーザーストーリー1.3（例：プロフィール編集）
├── エピック2（例：商品管理）
│   ├── ユーザーストーリー2.1（例：商品検索）
│   └── ユーザーストーリー2.2（例：商品詳細表示）
```

### 🎯 ユーザーストーリー形式

```
As a [ユーザータイプ]
I want [機能・行動]
So that [価値・目的]
```

### 📊 優先順位付け

| 方法 | 説明 |
|------|------|
| **MoSCoW法** | Must/Should/Could/Won't |
| **価値ベース** | ビジネス価値・ユーザー価値 |
| **リスクベース** | 技術リスク・市場リスク |
| **依存関係** | 前提条件・順序関係 |

## INVEST原則

| 原則 | 説明 | チェック |
|------|------|---------|
| **I**ndependent | 他のストーリーと独立 | ☐ |
| **N**egotiable | 詳細は交渉可能 | ☐ |
| **V**aluable | ユーザー価値がある | ☐ |
| **E**stimable | 見積もり可能 | ☐ |
| **S**mall | 1スプリントで完了可能 | ☐ |
| **T**estable | テスト可能 | ☐ |

## バックログ管理のベストプラクティス

### ✅ 良いバックログ

- INVEST原則に準拠
- ユーザー中心の価値記述
- 明確な優先順位付け
- 適切なサイズ（1スプリントで完了）

### ❌ 避けるべきバックログ

- 技術偏重（技術実装のみに焦点）
- 曖昧（要求が不明確）
- 巨大（1つのストーリーが大きすぎる）
- 依存過多（複雑な依存関係）

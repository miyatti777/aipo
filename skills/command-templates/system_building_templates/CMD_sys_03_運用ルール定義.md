# CMD_sys_03_運用ルール定義

最終更新日時: 2025年12月25日 9:06

# テンプレート概要

このテンプレートは、**運用ルールを体系化し、実際に動くルール管理システムを構築する**ためのものです。

## 📋 使用場面

- 新しい業務の運用ルールを策定する
- 暗黙知を形式知化する
- チーム・部門の運用ガイドラインを整備する
- 例外処理の判断基準を明確化する
- よくある質問（FAQ）を体系化する

- [ ]  運用ルールDatabaseが作成されている
- [ ]  ルールカテゴリが整理されている
- [ ]  各ルールに例外処理フローが定義されている
- [ ]  FAQ Databaseが作成されている
- [ ]  サンプルルール・FAQが5件以上登録されている
- [ ]  運用ガイドドキュメントが作成されている

# Phase 2: 運用ルールDatabaseの作成

## 🎯 実行内容

運用ルールを管理する**実際に動くDatabase**を作成する。

### 2.1 Database設計

**必須プロパティ：**

| プロパティ名 | 型 | 説明 |
| --- | --- | --- |
| ルール名 | title | ルールの名称 |
| カテゴリ | select | ルールの分類 |
| 重要度 | select | 高/中/低 |
| 対象範囲 | multi_select | 適用対象（部門・チーム等） |
| ルール内容 | text | 詳細な説明 |
| 例外条件 | text | 例外が認められる条件 |
| 例外承認者 | person | 例外を承認できる人 |
| 参考リンク | url | 関連ドキュメントへのリンク |
| ステータス | status | 有効/検討中/廃止 |
| 作成日 | created_time | ルール作成日 |
| 最終更新日 | last_edited_time | 最終更新日 |

**カテゴリオプション例：**

- 📋 業務フロー
- ⚖️ 判断基準
- 🔒 コンプライアンス
- 💡 ベストプラクティス
- ⚠️ 例外処理
- 🚫 禁止事項

**ステータスオプション：**

- ✅ 有効
- 🔄 検討中
- ⏸️ 一時停止
- ❌ 廃止

### 2.2 ビュー設計

**作成するビュー：**

1. **📋 全ルール一覧（Table）**
    - すべてのルールを一覧表示
    - グループ化：カテゴリ
2. **⭐ 重要ルール（Table）**
    - フィルター：重要度 = 高 AND ステータス = 有効
    - 必ず知っておくべきルール
3. **📂 カテゴリ別（Board）**
    - カラム：カテゴリ
    - カテゴリごとにルールを整理
4. **👥 部門別（Table）**
    - グループ化：対象範囲
    - 各部門向けのルール表示
5. **🔄 見直し対象（Table）**
    - フィルター：最終更新日 < 6ヶ月前
    - 定期見直しが必要なルール

# Phase 3: 例外処理フローの定義

## 🎯 実行内容

例外処理のフローを設計し、判断基準を明確化する。

### 3.1 例外処理フロー図

**作成する内容：**

```
## 例外処理フロー

### Step 1: 例外発生の認識

[通常ルールが適用できない状況]
  ↓
[例外処理の必要性を確認]
  ↓ YES
[例外申請へ]

### Step 2: 例外申請

[該当ルールを確認]
  ↓
[例外条件に該当するか確認]
  ↓ YES              ↓ NO
[承認者に申請]    [通常ルール適用]
  ↓
[承認待ち]

### Step 3: 承認判断

[承認者が状況を評価]
  ↓
[リスク・影響を確認]
  ↓
[承認 / 却下]
  ↓ 承認           ↓ 却下
[例外処理実行]   [代替案検討]
  ↓
[記録・報告]

### Step 4: 記録と改善

[例外処理ログに記録]
  ↓
[定期的にレビュー]
  ↓
[頻発する例外はルール改定を検討]
```

### 3.2 例外判断マトリクス

| 影響度  頻度 | 低頻度 | 中頻度 | 高頻度 |
| --- | --- | --- | --- |
| **高影響** | 個別承認 | ルール見直し | 即座にルール改定 |
| **中影響** | 個別承認 | 簡易承認 | ルール見直し |
| **低影響** | 簡易承認 | 簡易承認 | 標準化検討 |

**承認レベル定義：**

- **個別承認**：上位管理者の個別判断が必要
- **簡易承認**：現場責任者の判断でOK
- **ルール見直し**：例外処理委員会で検討
- **ルール改定**：新ルールとして標準化

# Phase 5: サンプルデータ投入

## 🎯 実行内容

実際のルールとFAQを想定したサンプルデータを投入する。

### 5.1 サンプルルール作成（最低5件）

**ルール例：**

1. **承認金額基準**
    - カテゴリ：判断基準
    - 内容：「100万円以上の支出は部長承認が必要」
    - 例外条件：「緊急の設備故障対応は事後承認可」
    - 重要度：高
2. **レビュー期限**
    - カテゴリ：業務フロー
    - 内容：「ドキュメントレビューは依頼から3営業日以内」
    - 例外条件：「大型連休前後は5営業日まで延長可」
    - 重要度：中
3. **データアクセス制限**
    - カテゴリ：コンプライアンス
    - 内容：「顧客情報へのアクセスは業務上必要な者のみ」
    - 例外条件：「例外なし（違反は懲戒対象）」
    - 重要度：高
4. **会議時間設定**
    - カテゴリ：ベストプラクティス
    - 内容：「会議は原則30分以内、最大60分」
    - 例外条件：「ワークショップ等は2時間まで可」
    - 重要度：低
5. **緊急連絡フロー**
    - カテゴリ：例外処理
    - 内容：「システム障害発生時は即座に責任者へ連絡」
    - 例外条件：「夜間・休日は当番責任者へ」
    - 重要度：高

### 5.2 サンプルFAQ作成（最低5件）

**FAQ例：**

1. **Q: 100万円ちょうどの場合も部長承認が必要？**
    - A: はい、100万円以上は部長承認が必要です。
    - 関連ルール：承認金額基準
2. **Q: レビュー期限を過ぎそうな場合はどうすればいい？**
    - A: 期限前に依頼者に連絡し、延長を相談してください。
    - 関連ルール：レビュー期限
3. **Q: 顧客情報を社外に持ち出せますか？**
    - A: 原則禁止です。必要な場合は情報セキュリティ部門の承認が必要です。
    - 関連ルール：データアクセス制限
4. **Q: 会議が延びそうな場合は？**
    - A: 30分経過時点で延長判断を行い、参加者の合意を得てください。
    - 関連ルール：会議時間設定
5. **Q: 夜間のシステム障害はどこに連絡？**
    - A: 当番責任者の緊急連絡先（社内Wikiに記載）に連絡してください。
    - 関連ルール：緊急連絡フロー

## 📋 このガイドの目的

このガイドは、[対象業務]の運用ルールと例外処理について説明します。

**対象者**：[ユーザー層]
**適用範囲**：[部門・チーム]
**最終更新**：[日付]

---

## 📚 運用ルールDatabase

<mention-database url="[運用ルールDB]"/>

### 使い方

1. **ルールを探す**
   - カテゴリ別ビューで分類から探す
   - 検索機能でキーワード検索

2. **ルールを理解する**
   - ルール内容を確認
   - 例外条件があるか確認
   - 関連ルールも参照

3. **例外処理が必要な場合**
   - 例外条件に該当するか確認
   - 例外承認者に相談
   - Phase 3の例外処理フローに従う

---

## ❓ FAQ Database

<mention-database url="[FAQ DB]"/>

   - [回答概要]

## ⚠️ 例外処理フロー

[Phase 3で作成したフロー図へのリンク]

### 例外処理の原則

1. **透明性**：例外処理は必ず記録する
2. **一貫性**：同じ状況では同じ判断をする
3. **学習**：頻発する例外はルール見直しのサイン
4. **承認**：承認者の判断を仰ぐ

---

## 🔄 ルールの見直しプロセス

### 見直し頻度

- **定期見直し**：四半期ごと
- **臨時見直し**：重大な問題発生時

### 見直し手順

1. **現状評価**
   - 例外処理の発生頻度を確認
   - ユーザーからのフィードバック収集

2. **改善検討**
   - 問題のあるルールを特定
   - 改善案を作成

3. **承認**
   - 責任者承認
   - 関係者への周知

4. **更新**
   - Databaseを更新
   - ステータスを「有効」に変更

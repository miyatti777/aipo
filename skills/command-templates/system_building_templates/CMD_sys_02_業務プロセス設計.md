# CMD_sys_02_業務プロセス設計

最終更新日時: 2025年12月25日 9:06

# テンプレート概要

このテンプレートは、**業務プロセスを設計し、実際に動くワークフローシステムを構築する**ためのものです。

## 📋 使用場面

- 新しい業務プロセスを立ち上げる
- 既存プロセスを標準化・可視化する
- 承認フロー、レビューフロー、問い合わせ対応フローなどを構築
- 部門間連携プロセスを定義する

- [ ]  プロセスフロー図が完成している
- [ ]  ワークフローDatabaseが作成され、ステータス遷移が定義されている
- [ ]  サンプルケースが3件以上登録され、動作確認済み
- [ ]  プロセス定義ドキュメントが作成されている
- [ ]  例外処理フローが定義されている
- [ ]  運用開始準備が完了している

# Phase 1: プロセス要件の確認

## 🎯 実行内容

ユーザーと対話し、以下を明確化する：

### 1.1 プロセスの基本情報

**AI質問項目：**

> 以下の情報を教えてください：
> 

> 
> 

> 1. **プロセス名**：このプロセスの名称は？
> 

> 2. **プロセスの目的**：何を達成するためのプロセスか？
> 

> 3. **対象範囲**：どの部門・チームが関わるか？
> 

> 4. **トリガー**：このプロセスはどのようなきっかけで開始されるか？
> 

> 5. **ゴール**：プロセスが完了した状態とは？
> 

### 1.2 ステップとステータスの定義

**AI質問項目：**

> プロセスのステップとステータスを教えてください：
> 

> 
> 

> 1. **主要ステップ**：プロセスは何段階に分かれるか？（例：申請→承認→実行→完了）
> 

> 2. **各ステップの担当者/ロール**：誰が何を担当するか？
> 

> 3. **判断ポイント**：承認/却下などの分岐はあるか？
> 

> 4. **所要時間**：各ステップの標準処理時間は？
> 

### 1.3 例外処理の定義

**AI質問項目：**

> 例外ケースについて教えてください：
> 

> 
> 

> 1. **想定される例外**：どのような例外が発生しうるか？
> 

> 2. **エスカレーション**：問題発生時の対応ルートは？
> 

> 3. **期限切れ対応**：処理が遅延した場合の対応は？
>

**AI → ユーザー：**

> 以下のプロセス設計で進めてよろしいですか？
> 

> 
> 

> **プロセス名**：[名称]
> 

> **ステータス遷移**：[開始] → [ステップ1] → [ステップ2] → ... → [完了]
> 

> **担当ロール**：[一覧]
> 

> **例外処理**：[概要]
> 

> 
> 

> 問題なければ「OK」、修正があれば詳細を教えてください。
>

# Phase 2: プロセスフロー図の作成

## 🎯 実行内容

プロセスフロー図をNotion上に作成する。

### 2.1 フロー図の構成

**作成する内容：**

```
## プロセスフロー図

### 通常フロー

[開始] 
  ↓
[ステップ1: 申請]
  - 担当: [ロール名]
  - 処理: [内容]
  - 所要時間: [時間]
  ↓
[判断: 承認可否]
  ↓ YES              ↓ NO
[ステップ2: 実行]   [却下]
  ↓
[完了]

### 例外フロー

[エスカレーション発生]
  ↓
[上位承認者へ]
  ↓
[特別対応]
```

### 2.2 ステータス遷移表

| 現在ステータス | 次ステータス候補 | 条件 | 担当ロール |
| --- | --- | --- | --- |
| 申請中 | 承認待ち | 申請完了 | 申請者 |
| 承認待ち | 承認済み / 却下 | 承認判断 | 承認者 |
| 承認済み | 実行中 | 作業開始 | 実行者 |
| 実行中 | 完了 | 作業完了 | 実行者 |

**AI → ユーザー：**

> プロセスフロー図を作成しました。
> 

> 内容を確認して、修正があれば教えてください。問題なければ「OK」で次に進みます。

# Phase 3: ワークフローDatabaseの作成

## 🎯 実行内容

プロセスを管理する**実際に動くDatabase**を作成する。

### 3.1 Database設計

**必須プロパティ：**

| プロパティ名 | 型 | 説明 |
| --- | --- | --- |
| ケース名 | title | 案件・チケットの名称 |
| ステータス | status | 現在のプロセス状態 |
| 担当者 | person | 現在の担当者 |
| 申請者 | person | プロセスを開始した人 |
| 申請日 | date | プロセス開始日 |
| 期限 | date | 完了目標日 |
| 優先度 | select | 高/中/低 |
| カテゴリ | select | プロセスの種類分類 |
| 説明 | text | 詳細内容 |

**ステータスオプション（例）：**

- 📝 申請中
- ⏳ 承認待ち
- ✅ 承認済み
- 🔄 実行中
- ⚠️ 保留
- ❌ 却下
- ✔️ 完了

### 3.2 ビュー設計

**作成するビュー：**

1. **📋 全件一覧（Table）**
    - すべてのケースを時系列表示
    - グループ化：ステータス
2. **👤 担当者別（Board）**
    - カラム：担当者
    - 各担当者のタスクを可視化
3. **📊 ステータス別（Board）**
    - カラム：ステータス
    - カンバン形式でプロセス進行を管理
4. **📅 期限管理（Calendar）**
    - カレンダー表示：期限
    - 期限切れリスクの可視化
5. **⚠️ 要注意（Table）**
    - フィルター：ステータス = 保留 OR 期限 < 今日
    - 対応が必要なケースのみ表示

**AI → ユーザー：**

> ワークフローDatabaseを作成しました：
> 

> [作成したDB]
> 

> 
> 

> プロパティとビューを確認してください。問題なければ「OK」で次に進みます。

# Phase 5: プロセス定義ドキュメントの作成

## 🎯 実行内容

プロセスの運用マニュアルとなるドキュメントを作成する。

### 5.1 ドキュメント構成

```markdown
# [プロセス名] 運用ガイド

## 📋 プロセス概要

**目的**：[プロセスの目的]
**対象範囲**：[関係部門・チーム]
**トリガー**：[開始条件]
**ゴール**：[完了状態]

---

## 🔄 プロセスフロー

[Phase 2で作成したフロー図へのリンク]

---

## 📊 ワークフローDatabase

<mention-database url="[作成したDB]"/>

---

## 👥 ロールと責任

| ロール | 責任 | 担当ステータス |
|-------|------|-------------|
| 申請者 | ケースの申請と情報提供 | 申請中 |
| 承認者 | 申請内容の承認判断 | 承認待ち |
| 実行者 | 承認された作業の実行 | 実行中 |
| 管理者 | プロセス全体の監視 | 全体 |

---

## 📝 各ステータスでの作業内容

### 申請中

**担当**：申請者
**作業内容**：
1. 新しいケースをDatabaseに作成
2. 必要情報を入力（説明、期限、カテゴリ等）
3. ステータスを「承認待ち」に変更
4. 承認者をメンション通知

### 承認待ち

**担当**：承認者
**作業内容**：
1. 申請内容を確認
2. 判断基準に基づき承認/却下を決定
3. ステータスを「承認済み」または「却下」に変更
4. コメントで理由を記載

### 実行中

**担当**：実行者
**作業内容**：
1. 承認済みケースを確認
2. 作業を実行
3. 進捗をコメントで更新
4. 完了後、ステータスを「完了」に変更

---

## ⚠️ 例外処理

### 追加情報が必要な場合

1. ステータスを「保留」に変更
2. 申請者にコメントで追加情報を依頼
3. 情報が揃ったら元のステータスに戻す

### 期限を過ぎた場合

1. 「要注意」ビューで期限切れを検知
2. 担当者に催促
3. 必要に応じてエスカレーション

### 承認が却下された場合

1. ステータスを「却下」に変更
2. 却下理由をコメントに記載
3. 申請者に通知
4. 必要に応じて再申請

---

## 📈 運用KPI

- **平均処理時間**：申請から完了までの平均日数
- **承認率**：承認されたケースの割合
- **期限遵守率**：期限内に完了したケースの割合
- **滞留ケース数**：「承認待ち」「保留」の件数

---

## 🔧 トラブルシューティング

### Q: ステータスが変更できない
A: 編集権限があるか確認。Database設定を確認。

### Q: 通知が来ない
A: Notion通知設定を確認。メンションが正しいか確認。

### Q: 古いケースが残っている
A: 定期的に完了ケースをアーカイブ。「完了」ステータスでフィルター。
```

## 💡 カスタマイズ例

### 承認フローの場合

- ステータス：申請中 → 1次承認待ち → 2次承認待ち → 承認済み → 完了
- 承認者を複数段階に分ける

### 問い合わせ対応の場合

- ステータス：新規 → 対応中 → 保留 → 解決済み → クローズ
- 優先度とカテゴリで分類

### レビュープロセスの場合

- ステータス：下書き → レビュー待ち → 修正中 → 承認済み → 公開
- レビュアーをpersonプロパティで管理

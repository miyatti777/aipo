# CMD_com_Slack依頼分析＆タスク化

最終更新日時: 2025年12月25日 9:06

## 📌 コマンド概要

**目的**: Slackで受けた依頼を構造化して理解し、戦略的な対応策を立案してFlowに配置する

**用途**: 複雑な依頼・複数の文脈が絡む依頼の整理と実行計画の作成

**実行時間**: 5〜10分

---

## 📥 Input

### 必須項目

- **Slack依頼内容** (必須): 依頼者からのメッセージ全文
- **依頼者** (必須): 依頼者の名前（例: kanさん）

### オプション項目

- **関連URL**: 依頼に関連するSlackスレッドやファイルパス
- **期限**: 依頼の期限があれば明記
- **補足情報**: その他の背景情報

### 入力例

```
@CMD_Slack依頼分析＆タスク化 を実行してください

依頼者: kanさん
依頼内容: 
「Masa さんと動いてる PF の計画、今日の案件分け分けの議論踏まえ FDE 持つことも意識したものにお願いします！
まだ出来てなくて恐縮ですが、経営観点でいつまでに、どんな状態まで生きたいかってのは言語化しようと思ってるので、それとぶつけてすり合わせたいと思っています」
```

## 📋 処理ロジック

### ステップ1: コンテキスト収集と依頼分析 (3〜5分)

**目的**: 依頼の背景・文脈を理解し、本質的な要求を特定する

**実行内容**:

1. **依頼文の構造化**
    - 依頼者の背景（役職、担当領域）を確認
    - 依頼文から以下を抽出：
        - 対象（何について）
        - 要求（何をしてほしいか）
        - 制約条件（いつまでに、どのレベルで）
        - 背景・理由（なぜ今この依頼か）
2. **ワークスペースコンテキスト検索**
    - 依頼文中のキーワードで関連ドキュメントを検索
    - 特に以下を重点的に検索：
        - 進行中のプロジェクト
        - 関連するロードマップ・計画文書
        - 過去の議事録や意思決定記録
        - 関連する専門用語の定義（例: FDE、PMF等）
3. **現状把握**
    - 該当プロジェクトの現在の進捗状況を確認
    - 既存の計画・ロードマップとの整合性を確認
    - 関連するタスクやマイルストーンを確認
4. **依頼の本質特定**
    - 表面的な要求と本質的な要求を区別
    - 依頼の優先度を判断
    - 他の依頼・プロジェクトとの関連性を整理

**アウトプット例**:

```
## 📊 依頼分析

### 依頼者情報
- 氏名: kanさん（EXPLAZA CTO）
- 依頼日時: 2025-12-10 14:00頃

### 依頼の構造化
- **対象**: PF（Platform）の計画・ロードマップ
- **要求**: FDEモデルを意識した計画への更新
- **制約**: 今日の案件分け分け議論を反映、経営観点との整合性
- **背景**: Partner/Platform体制移行、FDEモデルの導入検討

### 関連プロジェクト・文書
- AI Native Platform Wiki
- 中長期ロードマップ v1.0
- Platform成長戦略 v1.0
- FDEデプロイメントプロセス
- Partner↔Platform 連携ダッシュボード

### 現状
- Masaさんと年内目処でロードマップ・Phase分解を精緻化中
- HubSpotでヒアリング実施中、顧客像・フェーズの解像度向上中
- 既にFDEモデルの基礎文書は作成済み

### 依頼の本質
**表面**: ロードマップにFDEを追加する
**本質**: Platform事業の成長戦略において、FDEモデルをどう組み込み、いつまでにどのような体制・能力を持つべきかを明確にする
```

### ステップ2: 対応戦略の立案 (2〜3分)

**目的**: 依頼に対する最適なアプローチと具体的なアクションを設計する

**実行内容**:

1. **ゴール設定**
    - 依頼者が期待する成果物を明確化
    - 完成の定義を設定
2. **アプローチ方針の決定**
    - 既存文書の更新 vs 新規作成
    - 単独作業 vs 協業の判断
    - 段階的実施 vs 一括実施の判断
3. **必要なアクションの洗い出し**
    - 情報収集が必要な項目
    - 作成・更新が必要な文書
    - 確認・すり合わせが必要なステークホルダー
4. **リスク・制約の特定**
    - 時間的制約
    - 情報不足
    - 依存関係

**アウトプット例**:

```
## 🎯 対応戦略

### ゴール
- **成果物**: PF計画にFDEモデルを統合したロードマップ更新版
- **完成の定義**: 
  - FDEの役割・責任範囲が明確
  - Phase別のFDE体制が定義されている
  - 既存のDS/FDE連携と整合している
  - 経営観点での目標時期と到達状態が反映されている

### アプローチ方針
- **既存文書の更新**: 「中長期ロードマップ v1.0」をベースに更新
- **協業**: kanさん、Masaさんとの定期的なすり合わせ
- **段階的実施**: 
  1. FDE統合版ドラフト作成（今週）
  2. 経営観点の目標との整合確認（来週）
  3. 最終版確定（年内）

### 必要なアクション
1. **情報収集**
   - kanさんの「経営観点での目標」を確認
   - 今日の案件分け分け議論の結果を確認
   - Masaさんのヒアリング結果を確認

2. **文書作成・更新**
   - FDE統合版ロードマップのドラフト作成
   - Phase別FDE体制の定義
   - FDE採用計画の追加

3. **すり合わせ**
   - kanさんへのドラフト共有とFB収集
   - Masaさんとの整合性確認
   - 関係者への展開

### リスク・制約
- kanさんの「経営観点での目標」がまだ言語化中
- 今日の案件分け分け議論の詳細が不明
- 年内という時間的制約
```

### ステップ3: アクションプランの作成とFlow配置 (2分)

**目的**: 実行可能なタスクリストを作成し、今日の作業フォルダに配置する

**実行内容**:

1. **タスクの分解**
    - 大きなアクションを実行可能な単位に分解
    - 各タスクの所要時間を見積もり
2. **優先度付け**
    - 緊急度と重要度で優先順位を決定
    - 依存関係を考慮した順序を設定
3. **期限設定**
    - 各タスクの期限を設定
    - マイルストーンを明確化
4. **Flowへの配置**
    - 今日の日付フォルダ配下にページを作成
    - タスクリストをチェックボックス形式で記載
    - 関連文書へのリンクを追加

**アウトプット例**:

```
## ✅ アクションプラン

### 今日（2025-12-10）
```

## ⚠️ 注意事項

### 実行時の心得

1. **依頼の本質を見極める**: 表面的な要求だけでなく、なぜその依頼が来たのか、本当に解決すべき課題は何かを考える
2. **文脈を広く探す**: 依頼文のキーワードだけでなく、関連する領域も広く検索する
3. **既存の資産を活用**: ゼロから作るのではなく、既存の文書・プロセスを最大限活用する
4. **早めの確認**: 不明点や前提条件は早めに依頼者に確認する
5. **進捗の可視化**: Flowに配置することで、進捗を可視化し、依頼者も確認できるようにする

### エラーハンドリング

1. **依頼内容が不明確な場合**
    - 推測で進めず、依頼者に確認事項をまとめて質問する
2. **関連情報が見つからない場合**
    - 検索キーワードを変えて再検索
    - 依頼者に関連文書のURLを尋ねる
3. **期限が厳しい場合**
    - MVP（最小限の成果物）を定義
    - 段階的な提供を提案

## 💡 活用例

### 例1: 複雑な戦略依頼

```
依頼内容: 「来期のPF事業の成長戦略を、競合分析を踏まえて立案してほしい」

→ Step1で競合情報、既存戦略文書、市場分析を検索
→ Step2で差別化ポイント、注力領域を特定
→ Step3で戦略立案のタスクをPhase別に分解
```

### 例2: 緊急の対応依頼

```
依頼内容: 「明日の経営会議で使う資料、最新数字に更新しておいて」

→ Step1で該当資料を特定、最新数字の所在を確認
→ Step2で更新箇所と確認フローを設計
→ Step3で今日中に完了するタスクリストを作成
```

### 例3: 曖昧な改善依頼

```
依頼内容: 「○○のプロセス、もっと効率化できないかな」

→ Step1で現行プロセスを文書から把握、ボトルネックを検索
→ Step2で改善案を複数立案、効果とコストを比較
→ Step3で提案書作成とPoCのタスクを設定
```

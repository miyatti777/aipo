# 02 発見: 仮説マップ（前提条件マップ）

## 最初に質問（実行前に回答してください）

- プロジェクト名（必須）
- 主要な前提条件・仮説（必須）
- 仮説の根拠・情報源（任意）
- 検証方法・タイミング（任意）

## 目的

PMBOKディスカバリーフェーズの手法に基づき、プロジェクトの前提条件や仮説を整理したAssumption Map（前提条件マップ）を作成します。プロジェクト成功の前提となる仮説を明確化し、検証計画を策定します。

## 必要入力

- プロジェクト名: 対象プロジェクトの名称
- 前提条件: プロジェクト成功の前提となる条件・仮説
- 根拠: 仮説の根拠となる情報・データ・経験
- 検証方法: 仮説を検証する方法・手段
- 検証タイミング: いつ検証するか

## 仮説マップの構成要素

### 🎯 仮説の分類

| カテゴリ | 説明 | 例 |
|---------|------|-----|
| **市場仮説** | ターゲット市場・顧客ニーズに関する仮説 | 「この課題を持つユーザーは十分いる」 |
| **技術仮説** | 技術的実現可能性に関する仮説 | 「この技術で解決できる」 |
| **ビジネス仮説** | 収益性・事業性に関する仮説 | 「この価格で購入される」 |
| **組織仮説** | チーム・リソースに関する仮説 | 「このチームで開発できる」 |

### 📊 仮説の評価軸

| 軸 | 評価基準 |
|----|---------|
| **重要度** | プロジェクト成功への影響度（高/中/低） |
| **確実性** | 仮説の確からしさ（確実/普通/不確実） |
| **検証難易度** | 検証の難しさ（易/普通/難） |
| **検証コスト** | 検証にかかるコスト・時間 |

### 🔍 検証方法

- **調査**: アンケート・インタビュー・市場調査
- **実験**: プロトタイプ・MVP・A/Bテスト
- **分析**: データ分析・競合分析・技術調査
- **レビュー**: 専門家レビュー・ユーザーテスト

- **作成日**: [YYYY-MM-DD]
- **バージョン**: 1.0

## 仮説一覧

| ID | 仮説 | カテゴリ | 重要度 | 確実性 | 検証方法 | 検証時期 |
|----|------|---------|--------|--------|---------|---------|
| H001 | [仮説内容] | 市場 | 高/中/低 | 高/中/低 | [方法] | [時期] |
| H002 | [仮説内容] | 技術 | 高/中/低 | 高/中/低 | [方法] | [時期] |

## 仮説マトリクス

```
        確実性低     確実性高
重要度高 [要検証]     [前提]
重要度低 [監視]       [安全]
```

## 活用方法

### 🎯 リスク管理

- 仮説が外れた場合のリスクを事前に特定
- リスク対策の優先順位付け
- 早期警戒システムの構築

### 📋 プロジェクト計画

- 仮説検証のためのマイルストーン設定
- 検証結果に基づく計画調整
- Go/No-Go判断の基準設定

### 🔄 学習・改善

- 仮説と結果の比較による学習
- 次回プロジェクトへの教訓抽出
- 組織知識の蓄積

## トラブルシュート

- 仮説が抽象的すぎる場合は、具体的な検証可能な形に分解
- 検証方法が不明な場合は、類似プロジェクトの事例を参考
- 仮説が多すぎる場合は、重要度の高いものから優先的に整理
